# 三路相机只亮两路:USB 带宽与帧率钳制失效问题排查与修复

> 日期:2026-07-19
> 症状:相机绑定页三路相机(wrist / left_rear / right_rear)同时开启时,必有一路黑屏"等待首帧",任意两路组合都正常。同硬件下 lerobot 官方 `lerobot-record` 命令三路却都正常。
> 结论:**不是启动竞态,是 USB 等时带宽被前两路占满** —— 帧率钳制代码一直被驱动静默拒绝,相机实际按 60fps(而非 30fps)协商,单路带宽翻倍。已修复并真机验证。

## 1. 现象与环境

- 硬件:三台外置 UVC 相机(两台"USB摄像头" + 一台"icspring camera"),挂在同一 USB 控制器上(uniqueID 前缀 0x111/0x112/0x113 相邻)。
- 采集路径:`collect_studio/avf_capture.py` 的 `AVFCamStream`,AVFoundation `AVCaptureSession` 按 uniqueID 直采,640×480@30,输出 32BGRA。
- 对照组(正常):lerobot 官方命令,OpenCV 后端,wrist 640×480、两后摄 1280×720,`warmup_s: 3`。
- 项目内复现:任意两台相机都能正常出图,第三台(不固定是哪台)永远黑屏。

## 2. 排查过程(四个实验)

### 实验 0:枚举相机支持的格式(只读)

用 `AVCaptureDevice.formats()` 枚举,关键发现:

```
640x480 '420v'  fps <= [25, 30, 60]   # NV12 无压缩
640x480 'yuvs'  fps <= [10]           # YUY2 无压缩
1280x720 '420v' fps <= [25, 30, 60]
```

- **这批相机根本没有 MJPEG**(`mjpg`/`dmb1`),只有无压缩格式。640×480@30 NV12 ≈ 13.8MB/s/路。
- 同一 format 条目下帧率是离散档位 `[60-60] [30-30] [25-25]`。

### 实验 1:旧代码并行起三路(复现 bug)

```
wrist(0x111):      ok=False frames=0    err=等待首帧…
left_rear(0x112):  ok=True  frames=19   (~2fps,半死不活)
right_rear(0x113): ok=True  frames=499  (~55fps!远超目标的 30fps)
```

复现成功。注意 right_rear 跑出 ~55fps —— 说明 30fps 钳制没生效,为根因埋下伏笔。

### 实验 2:串行启动 + 等首帧(排除启动竞态)

逐台启动、每台等首帧稳定后再起下一台:

```
wrist:      首帧 0.5s,稳定 ~30fps
left_rear:  首帧 0.3s,稳定 ~60fps
right_rear: 0 帧(最后启动,一帧都收不到)
```

**结论:不是启动竞态**。谁后启动谁死,与设备无关 —— 典型的资源(带宽)竞争,前两路占满总线,第三路饿死。

### 实验 3:显式 setActiveFormat_ + CMTimeMake(1, 30) 钳帧率(失败,但揭开根因)

改用显式选格式 + `setActiveVideoMinFrameDuration_(CMTimeMake(1, 30))`,三路全部抛异常:

```
NSInvalidArgumentException -[AVCaptureDevice_Tundra setActiveVideoMinFrameDuration:]
Not supported - Supported ranges: ("<AVFrameRateRange 30.00 - 30.00
(1000000 / 30000030 - 1000000 / 30000030)>"), tried to set maxFrameRate to 30.000000 (1 / 30)
```

**这是本次排查最关键的一步**,它同时揭示了两件事:

1. UVC 驱动对帧时长做**精确匹配**:它支持的 CMTime 是 `1000000/30000030`(timescale 30000030),自造的 `1/30` 即使数值等价也会被拒。
2. **旧代码里同样的调用一直也在抛这个异常**,只是被 `try/except: pass` 吞掉了 → 帧率钳制从未生效 → session 按格式支持的最高帧率(60fps)协商 → 单路带宽直接翻倍(640×480@60 NV12 ≈ 27.6MB/s)。

### 实验 4:用 AVFrameRateRange 自带的 CMTime 钳帧率(成功)

从选中格式的 `videoSupportedFrameRateRanges()` 里取覆盖目标 fps 的 range,直接用它的 `minFrameDuration()/maxFrameDuration()`(即驱动认的那个精确 CMTime):

```
wrist:      fps=30.0
left_rear:  fps=25.0
right_rear: fps=25.0
```

**三路全部出图**。两路 60fps 能力的相机落在 25fps:总线带宽依然紧张,驱动选择优雅降帧而不是饿死某一路 —— 这是可接受的降级,30Hz 采集环不受影响(帧可重复采样)。

## 3. 根因总结

```
无压缩 640x480 NV12 @30fps ≈ 13.8MB/s/路,三路 ≈ 41MB/s(本就贴着 USB2 等时带宽上限)
        ×
帧率钳制静默失效(驱动拒绝自造 CMTime,异常被吞)→ 实际按 60fps 协商,单路翻倍至 ~27.6MB/s
        =
前两路(30+60 或 60+60fps)占满等时带宽 → 第三路一帧都收不到
```

**为什么 lerobot 正常?** 它的两台后摄跑 1280×720:无压缩 720p@30 ≈ 41.5MB/s 根本放不上总线,协商反而落在低带宽格式/低帧率上,三路总占用都很小。与 `warmup_s` 关系不大 —— warmup 的价值是"连接时等到首帧、失败即报错",并不解决带宽。

## 4. 修复方案

### 4.1 `collect_studio/avf_capture.py`

**(a) 新增格式选择辅助**:在设备格式列表中找精确匹配 640×480 且帧率档位覆盖目标 fps 的 `(format, frameRateRange)`;找不到返回 `(None, None)` 回退旧的 preset 协商。

**(b) `_start()` 建 session 之前显式设格式并钳帧率**(顺序很关键:session 一旦按 preset 启动就会重协商、覆盖设备设置):

```python
fmt, fps_range = _pick_format_and_fps_range(dev, self.width, self.height, self.fps)
if fmt is not None:
    ok, _ = dev.lockForConfiguration_(None)
    if ok:
        # UVC 驱动对帧时长做精确匹配,必须用 range 自带的 CMTime;
        # 自己 CMTimeMake(1, fps) 会被拒(实测抛 NSInvalidArgumentException)
        dev.setActiveFormat_(fmt)
        dev.setActiveVideoMinFrameDuration_(fps_range.maxFrameDuration())
        dev.setActiveVideoMaxFrameDuration_(fps_range.minFrameDuration())
        dev.unlockForConfiguration()
```

session 侧配合:显式设过 `activeFormat` 时把 preset 设为 `AVCaptureSessionPresetInputPriority`,告诉 session 不要再用 preset 覆盖设备格式;未匹配到格式时保持旧的 `AVCaptureSessionPreset640x480` 回退路径。

**(c) 串行启动 + warmup + 整路重试**(借鉴 lerobot 顺序 connect + warmup 的思路):

```python
_START_LOCK = threading.Lock()   # 同一时间只做一路的 USB 带宽协商
_WARMUP_S = 3.0                  # 每路启动后等首帧的最长时长
_START_ATTEMPTS = 3              # 收不到帧时整路重建的最多次数
```

`_start_safe()` 在 `_START_LOCK` 下:启动 → 等首帧(≤3s)→ 无帧则 `stopRunning` 重建重试,3 次都失败才置错"多次启动均未收到画面(可能 USB 带宽不足或设备被占用)"。

**(d) `stop()` 拆出 `_stop_session()`**:重试需要"停 session 但不置 `_stopped`",原 `stop()` 复用该助手。

**(e) 模块 docstring 记录实测结论**(带宽数字、60fps 协商、CMTime 精确匹配),防止后人回退。

### 4.2 `collect_studio/cams.py`

`_sync_stream` 原来见 `not ok and err` 就立即杀流重建。新流自带最长 ~10.5s 的启动重试,期间 err 只是"等待首帧" —— 加 **12s 年龄保护**,只重建确认僵死的流:

```python
if s and not s.ok and s.err and time.time() - getattr(s, "started_at", 0) > 12:
    s.stop(); del self.streams[uid]; s = None
```

`getattr(..., 0)` 兼容测试里的 `FakeStream`(无 `started_at` 属性,缺省视为"足够老",保持原测试语义:僵死流应被重建)。

- 真机(未起服务,直接调采集类):按 `CamManager.start_all` 的方式并行创建三路:


## 6. 经验与备忘

1. **macOS UVC 帧时长必须精确匹配**:设 `activeVideoMin/MaxFrameDuration` 要用 `AVFrameRateRange` 自带的 CMTime,不要 `CMTimeMake` 自造;离散档位(60/30/25)的设备尤其如此。
2. **"尽力而为"的 try/except pass 是隐患**:旧帧率钳制因此失效了数月无人发现。对这类调用要么真处理异常,要么至少 log 一次。
3. **多路 UVC 同开,带宽是第一嫌疑**:无压缩格式带宽 = 宽×高×像素字节×fps,帧率被协商翻倍时杀伤力翻倍;验证手段是看各路实际帧率,不是只看"有没有图"。
4. **症状"谁后启动谁死"≈资源竞争;"固定某台死"≈设备/接线问题**。先换启动顺序就能区分,省去大量弯路。
5. 带宽饱和但不出错时,macOS UVC 驱动会降帧(30→25)而不是断流 —— 可接受,但要知道自己的系统在红线附近。
6. 若未来要更高分辨率/帧率的多路采集,这批相机没有 MJPEG,只能靠降 fps 或换支持压缩格式的相机(或确认走 USB3 控制器)。
