# SO101 数据采集工作台(Collect Studio)计划书

> 目标一句话:在浏览器里完成「设备自检 → 一键校准导入 → 相机可视化绑定 → 翻书式批量采集 → v2.1 + EEF 数据落盘」全流程,键盘优先,坏数据零成本丢弃。

## 1. 设计原则

1. **绝不触碰舵机校准**:应用内没有任何调用 `calibrate()` 的路径,"校准"只做文件复制 + 健康检查。
2. **硬件身份用序列号持久化**:串口按 USB serial number 识别,相机按 AVFoundation uniqueID 识别,index/端口漂移自动重解析。
3. **先暂存、后入库**:录制写入 staging 目录,「保存」才晋升入库,「舍弃」= 删目录,不存在"录进数据集再想办法删"的问题。
4. **数据格式自己写 writer**:直接产出 v2.1(官方只有 v2.1→v3.0 单向转换脚本),同时注入 EEF 字段,不做 v3→v2 逆转换。
5. **旁挂不 fork**:后端 import lerobot 的设备类(`SO101Follower` / `SO101Leader` / `OpenCVCamera` / `RobotKinematics`),上游更新不受影响。
6. **键盘优先**:所有高频操作(开始/暂停/保存/舍弃/翻页/跳转/换任务)都有单键快捷键,鼠标只用于低频配置。

## 2. 总体架构

```
┌─────────────────────────── 浏览器(前端 SPA)───────────────────────────┐
│  ①设备与校准页   ②相机绑定页   ③采集台(翻书 deck + 任务面板 + 快捷键) │
└──────────────┬────────────────────────────────┬───────────────────────┘
        REST(控制/配置/episode 管理)      WebSocket(状态 + MJPEG 预览流)
┌──────────────┴────────────────────────────────┴───────────────────────┐
│                    后端 FastAPI(跑在 conda env `lerobot`)             │
│  DeviceManager      CameraManager        RecordService      Exporter  │
│  串口序列号识别      uniqueID↔index       30Hz 控制环         v2.1 writer│
│  校准文件复制/体检   多路抓帧线程          episode 状态机      EEF 注入   │
│                                          staging 落盘        校验器    │
└──────────────┬────────────────┬──────────────────┬────────────────────┘
          SO101Leader      SO101Follower      OpenCVCamera ×3
        (主动臂,只读)   (从动臂,写目标位)  (独立抓帧线程,取最新帧)
```

- **控制环与录制解耦**:teleop 线程常驻 30Hz(读主动臂 → 写从动臂 → 各相机取最新帧),录制状态只决定这一帧是否落盘。因此界面上任何时刻都能看到实时画面、随时开录。
- **前端技术选型**:Vite + React + TypeScript(也可纯 vanilla,demo 即 vanilla);预览流用 MJPEG-over-WebSocket(每路降到 ~10fps、480p 缩略图,不与录制抢带宽),历史回放直接 `<video>` 播已编码的 mp4(HTTP Range)。
- **部署**:`python -m collect_studio` 一条命令拉起,浏览器开 `http://localhost:8600`。

## 3. 模块设计

### 3.1 设备身份与一键校准(M1)

**串口识别**
- `serial.tools.list_ports` 枚举,记录 VID/PID/**serial number**。两臂控制板 VID/PID 相同,靠 serial number 区分。
- 首次绑定用**摆动识别**:界面提示"请晃动主动臂" → 后端同时轮询两个候选口的关节位置,位置在变的即 leader,另一个即 follower。全程零命令行、不会认反。
- 绑定结果写入 `configs/devices.yaml`,之后每次启动按 serial number 反查当前实际挂载的 `/dev/tty.usbmodem*`,拓展坞换口也不受影响。

**一键校准导入**
- 「导入校准」按钮 = 将 `configs/calibration_backup/` 内置的两份 json 复制到 `~/.cache/huggingface/lerobot/calibration/{robots/so_follower,teleoperators/so_leader}/<id>.json`(id 与 devices.yaml 一致)。换新机器、重装系统都是一键还原。
- 复制后做**校准体检**(只读):连接总线,读各关节当前位置,校验落在 `[range_min, range_max]` 内、homing_offset 非零;界面上逐关节亮绿灯/红灯。
- UI 上没有"重新校准舵机"入口;体检异常时提示人工检查接线/关节位置,而不是提供重校准按钮。

**急停**:界面常驻红色急停按钮 + `E` 键 → follower `disable_torque()`。

### 3.2 相机发现与可视化绑定(M2)

- **枚举**:macOS 上 OpenCV(AVFoundation 后端)的 index 顺序 = 系统 AVFoundation 设备顺序。用 `system_profiler SPCameraDataType -json`(或 pyobjc `AVCaptureDevice.uniqueID`)拿每台相机的 **uniqueID**,按枚举顺序建立 `uniqueID ↔ opencv index` 映射。
- **绑定 UI**:所有检测到的相机同屏出实时缩略图(不用再对着终端拍照猜)。每张卡片下三个角色按钮「腕部 / 左后 / 右后」,点选即绑定(同角色互斥);同时锁定分辨率/fps 配置。保存进 `devices.yaml`(按 uniqueID,永不按 index)。
- **启动自检**:某 uniqueID 找不到(没插/换了 hub 口)→ 该角色红牌告警并拒绝开录,而不是采到一半发现视角错位。

`configs/devices.yaml` 示例:

```yaml
arms:
  follower: { serial: "5B415327841", id: my_awesome_follower_arm }
  leader:   { serial: "5B415328621", id: my_awesome_leader_arm }
cameras:
  wrist:      { unique_id: "0x14200000...", width: 640,  height: 480, fps: 30 }
  left_rear:  { unique_id: "0x14210000...", width: 1280, height: 720, fps: 25 }
  right_rear: { unique_id: "0x14220000...", width: 1280, height: 720, fps: 25 }
record: { fps: 30, video_codec: h264_videotoolbox }
```

### 3.3 采集控制环与暂存(M3)

**录制状态机**

```
IDLE ──Space──▶ RECORDING ──Space──▶ PAUSED ──Space──▶ RECORDING
                   │  ▲                  │
              Enter│  └──────────────────┤Enter
                   ▼                     ▼
                 SAVE(晋升入库)      SAVE
                   │Backspace            │Backspace
                   ▼                     ▼
                 DISCARD(整目录删除,零成本)
```

**帧落盘策略**(与 lerobot-record 内部一致但更简单可控):
- 数值(observation.state / action / timestamp)追加进内存 buffer;
- 图像由每相机独立的异步 writer 线程写 JPEG 到 `staging/<session>/episode_xxx/frames/<cam>/%06d.jpg`,录制环不被磁盘 IO 卡顿;
- **保存**:ffmpeg 把 JPEG 序列编成 mp4(macOS 用 `h264_videotoolbox` 硬编,几秒内完成;AV1 软编在 mac 上太慢,不采用),数值写成 parquet,连同 `episode_meta.json`(任务、时长、帧数、相机快照、operator)一起晋升到 library;
- **舍弃**:`rm -rf` staging 里这个 episode 目录,结束。
- **暂停**:暂停期间不落帧,恢复后时间戳连续拼接(timestamp 用"已录制时长"而不是墙钟)。

### 3.4 采集台前端 —— 翻书式交互(M4)

**页面模型**:deck = `[ep_0, ep_1, …, ep_{N-1}, LIVE]`,最右永远是 LIVE 页。

- **LIVE 页**:腕部大图 + 两路后视小图,叠加 REC 指示灯、计时、帧数、当前任务提示词;
- **历史页**:直接 `<video>` 回放该 episode 的三路 mp4(带进度条),叠加元信息(任务、时长、帧数、保存时间);页内可一键**标废**(移入回收站,不物理删,可恢复;回收站支持一键清空);
- **导航**:`←/→` 翻页(带翻页动画),`Home/End` 跳最早/LIVE,`G` 弹出跳转框输 episode 号,底部胶片条(每集一格,颜色区分 已存/标废/LIVE)点击直达;
- **防误触**:录制中按 `←` 翻走会先自动暂停并提示。

**快捷键总表**(全部有对应 REST 接口,前端只是薄壳):

| 键 | 动作 |
|---|---|
| `Space` | 开始 / 暂停 / 恢复录制 |
| `Enter` | 保存当前 episode |
| `Backspace` | 录制中:舍弃;历史页:标废/恢复 |
| `← / →` | 上一页 / 下一页 |
| `Home / End` | 第一集 / 回到 LIVE |
| `G` | 跳转到指定 episode |
| `T` | 切换任务(轮采) |
| `E` | 急停(follower 断力矩) |
| `?` | 快捷键帮助 |

### 3.5 任务与批次管理(M5)

**三层概念**:
- **任务 task** = 语言提示词 + slug(如 `grab_cube` / "Grab the cube");
- **批次 session** = 某次连续采集(自动命名 `2026-07-17_1530_grab_cube`),绑定当时的相机配置快照与 operator;
- **episode** = 一条演示。

**目录布局**(原始数据与导出物分离,轮采/混采都不乱):

```
~/so101_data/
├── staging/<session>/episode_*/          # 录制暂存(掉电可恢复,舍弃即删)
├── library/<task_slug>/<session>/episode_*/   # 已保存的原始 episode 包(parquet+mp4+meta)
│                                              # 单集自包含,可随意增删,不影响别的集
├── trash/                                # 回收站(标废的 episode 移到这里)
└── exports/<dataset_name>/               # 打包产出的 v2.1 数据集(见 §3.6)
```

- **轮流采集**:采集台左侧任务面板随时 `T` 键切换任务,新 episode 自动挂到当前任务;各任务实时计数(今天已采 N 条)。
- **导出自由组合**:导出页勾选若干 task / session → 打包。既支持"单任务一个数据集",也支持"多任务合集"——v2.1 原生支持多任务(`tasks.jsonl` + 每帧 `task_index`),推荐语言条件 VLA 训练直接用多任务合集。
- episode 在 library 里是自包含的,**打包是纯读操作**,可反复以不同组合导出,原始数据永远不动。

### 3.6 v2.1 导出 + EEF 字段注入(M6)

**为什么自己写 writer**:官方仓库只有 `convert_dataset_v21_to_v30.py`(单向升级);且我们要加自定义特征,自己控制 parquet/meta 生成最干净。episode 包里的数据已经是"帧对齐的数值 + mp4",打包只是重排 + 编号 + 写 meta。

**v2.1 目录结构**(严格对齐 lerobot v2.1 规范):

```
exports/<name>/
├── meta/
│   ├── info.json            # codebase_version: "v2.1",features 含 eef 字段
│   ├── episodes.jsonl
│   ├── tasks.jsonl          # 多任务:每行 {task_index, task}
│   └── episodes_stats.jsonl
├── data/chunk-000/episode_000000.parquet …
└── videos/chunk-000/observation.images.{wrist,left_rear,right_rear}/episode_000000.mp4 …
```

**新增特征**:

> 2026-07-18 统一改造:导出层列名与 EEF 格式已与 so101-nexus 仿真采集对齐,
> 下面是**当前**规格(本节原先的四元数 + 增量方案已废弃)。内部原始 parquet 列名不变。

```json
"observation.pos_state": { "dtype": "float32", "shape": [6], "names": [...motors] },
"pos_action": { "dtype": "float32", "shape": [6], "names": [...motors] },
"observation.eef_state": { "dtype": "float32", "shape": [7],
  "names": ["x","y","z","roll","pitch","yaw","gripper"] },
"eef_action": { "dtype": "float32", "shape": [7],
  "names": ["x","y","z","roll","pitch","yaw","gripper"] }
```

**计算方法**:
1. 归一化关节值 → 弧度:按校准 json 的 `range_min/max` 反算 ticks(2048 ticks = URDF 零位)→ 角度;gripper 不参与 FK。
2. `observation.eef_state[t]` = `[FK(follower 关节角[t]) 的平移 x,y,z, roll, pitch, yaw, states[t][5]]`。
3. `eef_action[t]` = `[FK(leader action 关节角[t]) 的平移与欧拉角, actions[t][5]]`——
   **绝对目标位姿**,不是增量,也不从 state 序列差分。
4. **欧拉角约定**:RPY(extrinsic xyz),`R = Rz(yaw)·Ry(pitch)·Rx(roll)`,取值 `[-π, π]`;
   与 `fk.py:_rpy_to_R` 互逆(实测互逆误差 1.1e-15)。已知局限:`|pitch|→90°` 有万向节退化、
   `yaw` 跨 `±π` 有跳变。
5. gripper 通道与该数据集关节第 6 维同值同单位(0–100 归一化),不做换算。
6. URDF 用 SO-ARM100 官方 `so101_new_calib.urdf`,末端 frame `gripper_frame_link`;纯 numpy FK,无 placo 依赖。
7. 同时生成 GR00T `meta/modality.json`,所有条目显式写 `original_key`(不依赖 loader 默认值)。

**导出后自动校验**(写入 `meta/validation_report.json`,并显示在导出页):每集 parquet 行数 = 各 mp4 帧数;
timestamp 等差 1/fps;欧拉角 `|rpy| ≤ π`(`eef_euler_abs_max`);EEF gripper 通道与关节第 6 维逐帧相等
(`eef_gripper_err_max`,应为 0)。此外落盘 `eef_action_state_gap_max` =
`max |eef_action[t][:3] − eef_state[t+1][:3]|`——遥操作下 follower 跟随 leader 本就有滞后,
该项**只作诊断、不参与 `ok` 判定**。抽一集用旧版 LeRobotDataset(v2.1 代码)冒烟加载。

## 4. 关键风险与对策

| # | 风险 | 对策 |
|---|---|---|
| 4.1 | 相机 index / 串口路径漂移 | 一律按 uniqueID / serial number 持久化,启动时重解析;缺设备红牌拒录 |
| 4.2 | 三路 USB 相机同 hub 带宽 | 录制配置与现状一致(已验证可跑);前端预览流独立降采样(~10fps 缩略图),不影响录制抓帧 |
| 4.3 | 编码耗时卡录制 | 录制期只写 JPEG,编码放在「保存」时用 videotoolbox 硬编;保存期间可继续录下一集(编码在后台队列) |
| 4.4 | wrist 30fps 与 rear 25fps 时间轴 | 控制环 30Hz 为主时钟,rear 取最新帧(与 lerobot-record 行为一致,会有少量重复帧);info.json 统一写 fps=30。若在意可把三路统一 25fps,做成配置项 |
| 4.5 | v2.1 与新版 lerobot 读取不兼容 | 训练侧需用支持 v2.1 的 lerobot 版本(≤0.3.x)或自写 Dataset;导出器保证格式严格合规并附校验报告 |
| 4.6 | placo 安装失败 | 备选纯 numpy FK;M6 开工时先验证 `pip install placo` 在 conda env 里可用 |
| 4.7 | 录到一半崩溃 | staging 目录含已写 JPEG + 增量 jsonl,重启后提示"发现未完成 episode:恢复为可保存状态 / 丢弃" |
| 4.8 | 误删 | 界面无物理删除;只有回收站 + 手动清空回收站二次确认 |

## 6. Demo 说明(`demo.html`)

同目录 `demo.html` 是**纯前端交互稿**(无后端、单文件、可直接双击打开),用于确认交互后再动工:

- 完整走一遍三步流程:设备与校准(含摆动识别、一键导入动画)→ 相机绑定(模拟三路画面点选角色)→ 采集台;
- 采集台的翻书 deck、全部快捷键(Space/Enter/Backspace/←→/G/T/E/?)、任务轮采、胶片条、回收站均可真实交互;
- 相机画面为 canvas 模拟(机械臂 + 方块动画),真实版替换为 MJPEG 流与 mp4 回放,布局与交互不变。

看完 demo 若交互方向确认,即按 §5 里程碑动工。
